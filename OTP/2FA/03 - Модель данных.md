# Модель данных

## Основные сущности

### Tenant

Нужен, если система должна обслуживать несколько заказчиков или несколько систем внутри одного заказчика.

Поля:

- `id`
- `name`
- `status`
- `deployment_mode`
- `created_at`

### ApplicationClient

Система, которая интегрируется с вашим `2FA Server`.

Поля:

- `id`
- `tenant_id`
- `name`
- `client_type`
- `auth_method`
- `client_secret_hash`
- `redirect_uris`
- `allowed_factors`

### User

Пользователь второго фактора.

Поля:

- `id`
- `tenant_id`
- `external_subject_id`
- `username`
- `email`
- `phone`
- `status`

### FactorEnrollment

Факт подключения фактора к пользователю.

Поля:

- `id`
- `user_id`
- `factor_type`
- `status`
- `created_at`
- `confirmed_at`
- `last_used_at`

### TotpSecret

Поля:

- `enrollment_id`
- `secret_ciphertext`
- `secret_kek_version`
- `digits`
- `period_seconds`
- `algorithm`

Секреты должны храниться только в зашифрованном виде.

### Device

Поля:

- `id`
- `user_id`
- `platform`
- `device_name`
- `push_token`
- `public_key`
- `attestation_status`
- `last_seen_at`
- `revoked_at`

### Challenge

Центральная сущность подтверждения.

Поля:

- `id`
- `tenant_id`
- `application_client_id`
- `user_id`
- `operation_type`
- `factor_type`
- `status`
- `expires_at`
- `approved_at`
- `denied_at`
- `correlation_id`

### ChallengeAttempt

Поля:

- `id`
- `challenge_id`
- `attempt_type`
- `result`
- `ip`
- `user_agent`
- `created_at`

### BackupCode

Поля:

- `id`
- `user_id`
- `code_hash`
- `used_at`

### AuditEvent

Поля:

- `id`
- `tenant_id`
- `event_type`
- `actor_type`
- `actor_id`
- `subject_id`
- `severity`
- `payload_json`
- `created_at`

## Хранилища

### PostgreSQL

Постоянные данные:

- пользователи
- enrollment факторов
- устройства
- клиенты интеграций
- аудит

### Redis

Краткоживущие данные:

- активные `challenge`
- anti-replay ключи
- rate limit counters
- session state

## Правила безопасности данных

- не хранить `OTP` и `backup codes` в открытом виде
- секреты `TOTP` шифровать `DEK/KEK` схемой
- ключи шифрования выносить в `Vault` или `KMS`
- все критичные события писать в аудит без возможности тихого удаления
