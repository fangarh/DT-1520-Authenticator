# MVP и Roadmap

## MVP

Цель `MVP`: дать заказчику рабочий второй фактор для существующей системы без тяжелого `SSO`-ландшафта.

Состав:

- `2FA Core`
- `REST API`
- `TOTP`
- `Push approval`
- `Backup codes`
- `Admin UI`
- `PostgreSQL`
- `Redis`
- мобильное приложение
- базовый аудит

## Что не брать в первый релиз

- `SAML`
- `RADIUS`
- risk scoring на базе `ML`
- сложную федерацию идентичностей
- многоуровневые workflow подтверждения

## Этапы

### Этап 1. Foundation

- модель пользователей и клиентов интеграции
- выпуск `challenge`
- проверка `TOTP`
- аудит и базовые политики

### Этап 2. Mobile factor

- привязка устройства
- `push challenge`
- биометрическое подтверждение
- отзыв устройства

### Этап 3. Productization

- админка
- self-service enrollment
- backup codes
- Docker-поставка
- метрики и алерты

### Этап 4. Enterprise

- `OIDC`
- `LDAP/AD`
- `SIEM` export
- `HA` deployment

## Ключевые риски

- попытка сделать сразу и `IdP`, и `2FA provider`, и enterprise gateway
- завязка критического сценария только на `push`
- хранение секретов без нормальной схемы шифрования
- отсутствие стабильного интеграционного контракта

## Рекомендуемый старт

Если нужен реалистичный старт проекта, я бы шел так:

1. Начать с `REST-first` архитектуры.
2. Сделать сервер как отдельный продукт, не встраивая бизнес-логику заказчика.
3. Поддержать `TOTP + Push + Backup codes`.
4. Спроектировать все так, чтобы `OIDC` и коробка добавлялись без переделки ядра.

## Следующий практический шаг

После этой стадии уже можно делать:

- `ERD`
- `OpenAPI` для `v1`
- список микросервисов или решение в виде модульного монолита
- план реализации на 8-12 недель
