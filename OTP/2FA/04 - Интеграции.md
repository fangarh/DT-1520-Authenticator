# Интеграции

## Подход

Сначала стоит сделать один надежный интеграционный слой через `REST API`, а затем на его основе наращивать протоколы и адаптеры.

## Первая очередь интеграций

### REST API

Подходит для:

- собственных веб-приложений
- мобильных backend-систем
- внутренних корпоративных систем

Базовые методы:

- `POST /api/v1/challenges` создать второй фактор
- `GET /api/v1/challenges/{id}` получить статус
- `POST /api/v1/challenges/{id}/verify-totp` проверить код
- `POST /api/v1/challenges/{id}/approve` подтвердить challenge
- `POST /api/v1/challenges/{id}/deny` отклонить challenge
- `POST /api/v1/enrollments/totp` начать подключение `TOTP`
- `POST /api/v1/devices/activate` привязать смартфон

### SDK

Имеет смысл сделать минимум один официальный `SDK`, чтобы интеграция была не через голый `HTTP`.

Стартовые кандидаты:

- `.NET`
- `Java`
- `Node.js`

## Вторая очередь интеграций

### OIDC

Когда система созреет, можно выступать как `OIDC`-провайдер второго фактора или step-up провайдер.

Подходит для:

- современных `SSO`
- централизованной аутентификации
- порталов с несколькими приложениями

### SAML

Нужен для корпоративного сегмента и старых enterprise-систем.

### RADIUS

Нужен, если в прицеле:

- `VPN`
- сетевое оборудование
- Wi-Fi
- legacy-инфраструктура

### LDAP / AD Adapter

Не как основной протокол выдачи факторов, а как источник пользователей и групп.

## Рекомендуемый контракт интеграции

Внешняя система должна уметь:

1. передать идентификатор пользователя после первого фактора
2. получить `challenge_id`
3. опрашивать статус или получать webhook
4. принять решение о выдаче сессии только после статуса `approved`

## Webhooks

Для зрелой интеграции стоит предусмотреть:

- `challenge.approved`
- `challenge.denied`
- `challenge.expired`
- `device.enrolled`
- `factor.revoked`

## Версионирование

Сразу вводить `/api/v1`, потому что интеграционный слой будет жить дольше внутренней реализации.
