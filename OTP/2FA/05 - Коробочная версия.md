# Коробочная версия

## Цель

Сделать поставку, которую можно установить в контур заказчика без обязательной зависимости от внешнего облака.

## Формы поставки

### Базовая

- `Docker Compose`
- один узел `API`
- один `PostgreSQL`
- один `Redis`

Подходит для пилота и среднего заказчика.

### Enterprise

- `Kubernetes` + `Helm`
- несколько реплик `API`
- внешний `PostgreSQL` в `HA`
- отказоустойчивый `Redis`
- внешний `Vault/KMS`

## Компоненты коробки

- `API Gateway`
- `2FA Core`
- `Admin UI`
- `PostgreSQL`
- `Redis`
- `Push adapter`
- `Audit exporter`

## Что особенно важно для on-prem

- интеграция с `AD/LDAP`
- резервное копирование
- экспорт аудита в `SIEM`
- поддержка `SMTP`, прокси, внутренних сертификатов
- разграничение ролей администраторов
- обновление без потери секретов и enrollments

## Ограничения

Для коробки надо заранее решить:

- можно ли использовать внешние `FCM/APNs`
- нужен ли полностью изолированный режим без интернета
- кто управляет сертификатами и ключами шифрования

Если нужен полностью закрытый контур, `push` может работать хуже или не работать совсем. Тогда основным вторым фактором останется `TOTP`, а `push` будет опциональным.

## Модель лицензирования

Варианты:

- по числу активных пользователей
- по числу защищаемых приложений
- по числу tenant-ов
- per-instance для коробки

## Практический вывод

Архитектуру нужно сразу строить так, чтобы облачная и коробочная версии отличались конфигурацией, а не кодовой базой.
