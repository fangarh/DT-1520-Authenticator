# ADR-014: 2FA Server as MVP and IdP as Strategic Target

## Status

Accepted

## Context

Проекту нужен узкий и реализуемый `MVP`, который можно быстро интегрировать в существующие системы. Одновременно продукт не должен архитектурно замыкаться в роли только второго фактора, если долгосрочная цель состоит в переходе к полноценному `IdP`.

Если эту цель не зафиксировать заранее, есть риск спроектировать `User`, `ApplicationClient`, `Policy`, `Audit` и интеграционный слой так, что переход к identity-сценариям потом потребует дорогой переделки.

## Decision

- первый релиз остается отдельным `REST-first 2FA Server`
- стратегическая цель продукта фиксируется как эволюция в полноценный `IdP`
- `IdP`-возможности не входят в `MVP` и не расширяют текущий delivery scope
- архитектура `MVP` должна сохранять границы, которые позволят позже добавить primary auth, federation, session management и более широкий identity lifecycle

## Consequences

- `MVP` сохраняет узкий и безопасный scope
- будущие `OIDC`, `SAML`, directory и identity-сценарии становятся частью стратегического направления, а не случайным расширением
- backend и vault должны разделять текущую роль `2FA Server` и будущую роль `IdP`
- при проектировании доменной модели нельзя делать допущения, которые жестко привязывают продукт только к second-factor use cases
