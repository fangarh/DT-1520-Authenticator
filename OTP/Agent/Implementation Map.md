# Implementation Map

## Назначение

Этот файл нужен, чтобы новая сессия быстро понимала, где искать реальные артефакты, не сканируя весь репозиторий.

## Текущее содержимое репозитория

- `OTP/` — knowledge vault проекта и основная точка входа для контекста
- `backend/` — основной backend runtime на `.NET`
- `mobile/` — Android workspace на `Kotlin` с локально закрытыми `TOTP-first` и `push runtime` slices
- `admin/` — runtime `Admin UI MVP` на `React + Vite`
- `installer-ui/` — отдельный local setup shell на `React + Vite` с loopback-only bridge к installer engine
- `infra/` — packaging-корень для `on-prem` runtime contour и installer-артефактов
- `docs/` — standalone `React + Vite` documentation app без backend dependency
- `lib/` — `.NET` NuGet SDK workspace for `Dt1520.Authenticator.*`
- `rdb_stand/` — reference `Desktop + Backend` stand
- `config/mcp/` — локальные примеры MCP-конфигов

## Known Local Verification Workarounds

- Backend tests in this Windows workspace can hang or fail with `Access denied` on `backend/artifacts/obj` or `backend/artifacts/bin` because stale `dotnet`/`MSBuild` processes keep file locks. First inspect `Get-Process dotnet, MSBuild, VBCSCompiler -ErrorAction SilentlyContinue`, stop only stale processes from the failed run, then retry through `backend/scripts/verify-backend.ps1` or targeted `dotnet test ... -p:BuildInParallel=false -p:RestoreBuildInParallel=false -maxcpucount:1`.
- If backend verification fails only inside sandbox with `Access denied`, repeat the identical command outside sandbox with escalation instead of changing the test scope.
- Android/Gradle tests should run with `JAVA_HOME='C:\Program Files\Android\Android Studio\jbr'`. If Gradle hangs or fails on `%USERPROFILE%\.gradle\...\*.lck`, inspect `Get-Process java, gradle -ErrorAction SilentlyContinue`, stop only stale processes from the interrupted run, and retry the same Gradle command.
