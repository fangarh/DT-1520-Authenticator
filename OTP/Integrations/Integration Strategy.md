# Integration Strategy

## Подход

Первый релиз строится вокруг стабильного `REST API`. Протоколы уровня `OIDC`, `SAML` и `RADIUS` добавляются позже как отдельные адаптеры и режимы работы на пути к более широкому `Identity mode`.

## Первая очередь

- `REST API`
- официальный `SDK` хотя бы для одной платформы
- `webhook`-события по ключевым состояниям `challenge`
- `OAuth 2.0 client credentials` для интеграционных клиентов

Стартовые `SDK`-кандидаты:

- `.NET`
- `Java`
- `Node.js`

## Базовый контракт

Внешняя система должна уметь:

1. передать пользователя после успешного первого фактора
2. создать `challenge`
3. дать пользователю способ завершить `challenge`
4. получить итоговый статус
5. выдать сессию только после `approved`

## Базовые сценарии API

- создание `challenge`
- опрос статуса
- проверка `TOTP`
- подтверждение или отказ по `push`
- старт enrollment для `TOTP`
- активация устройства
- выпуск и обновление токенов

## Аутентификация

Интеграционные клиенты:

- `POST /oauth2/token`
- bearer token
- scope-ы по типу операций

Мобильные устройства:

- токены выдаются при активации устройства
- refresh идет через отдельный endpoint

Подробнее: [[Auth and Token Flows]]

## Вторая очередь

- `OIDC`
- `SAML`
- `RADIUS`
- `LDAP/AD` adapter

## Версионирование

- использовать `/api/v1`
- не ломать семантику существующих endpoint-ов внутри версии
- расширять через новые поля и новые ресурсы
